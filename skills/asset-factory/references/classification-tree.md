# Classification Tree — 3 분류 질문 상세

사용자의 요구사항을 **skill / command / team / agent / (생성 불필요)** 중 하나로 분류하는 decision tree입니다. Phase 1의 Q1~Q3 질문을 해석하는 상세 가이드.

## Scoped Pre-scan 프로토콜

**Step 1a(요구사항) 후에만 실행.** 요구사항이 없는 상태에서 broad scan 금지.

### 토큰 기반 scoped 스캔

1. Step 1a 요구사항 문자열에서 **명사·동사 토큰 추출** (불용어 제외)
2. 아래 Glob으로 자산의 **description/요약만** 수집 (경량 — 각 자산 3~5줄):
   - `~/.claude/skills/*/SKILL.md` → frontmatter description
   - `./.claude/skills/*/SKILL.md` → frontmatter description
   - `~/.claude/commands/*.md` → 파일명 + `<invocation>` 또는 첫 3줄
   - `./.claude/commands/*.md` → 동일
   - `./.claude/agents/*.md` → 존재 여부 + 개수
3. 각 자산 description 토큰과 요구사항 토큰의 **공통 비율** 계산
4. **30% 이상 겹치는 자산만** 사용자에게 노출 (나머지는 보고 생략)

### 왜 scoped인가

- Broad scan은 자산 10~30개 모두의 description을 사용자에게 나열하게 되어 시끄럽고 산만.
- 요구사항 토큰 기반 필터링으로 **정말 중복 위험이 있는 자산만** 보여주면 사용자가 (a)/(b)/(c) 결정이 쉬움.
- 스캔 자체 비용은 동일(모든 description을 읽긴 함)이나 **사용자에게 보여주는 부분**이 노이즈 감소.

### 중복 발견 시 안내

```
🔎 유사 자산 발견
- `<name>` (<kind>): <description 1줄>

(a) 이 자산을 확장·편집
(b) 별도로 새 자산 생성 (계속 진행)
(c) 이 자산을 먼저 확인하고 다시 결정
```

- (a) → Phase 2로 바로, 단 "새 생성" 대신 "편집 가이드"로 분기
- (b) → Step 1c(분류 질문) 계속
- (c) → 대화 종료

### 중복 없음 시

```
✅ 유사 자산 없음. 분류 질문으로 진행합니다.
```

바로 Step 1c로.

## Q1. 반복 가능한 워크플로인가?

### 목적

사용자가 **진짜 반복할 작업**을 만드는지, 지금 **한 번 도와달라는 요청**을 스킬로 만들려 하는지 구분. 후자는 생성 자체가 낭비.

### 답변 분류

- (a) **반복함** — 매번 같은 규칙·절차 필요.
- (b) **이번 한 번만** — 지금 작업이 끝나면 다시 쓸 일 없음.
- (c) **잘 모르겠음**

### (c) 답변 시 힌트

> *"다음 중 어느 쪽에 가깝나요?*
> *- 이 규칙은 이 프로젝트 / 이 팀 / 내 개발 습관에 매번 적용되어야 한다 → 반복함*
> *- 지금 이 순간 이 코드베이스의 특정 문제 해결이 목적이다 → 한 번만"*

### (b) 처리

생성 중단:
```
이 작업은 일회성 같습니다. 지금 바로 저(Claude)에게 요청하시는 편이 효율적입니다.
예: "<사용자의 요구사항>을 지금 처리해줘"

스킬·커맨드 생성은 같은 패턴을 **3회 이상 반복할 것이 확실할 때** 가치가 있습니다.
```

사용자가 그래도 만들고 싶다고 하면 존중하고 Q2로 진행 (자기 판단권).

### (a) 처리

Q2로 진행.

## Q2. 여러 전문가가 협업해야 하는가?

### 목적

단일 자산(skill/command/agent) vs 팀(harness-factory) 구분.

### 판단 기준

- **팀 필요**: 아래 중 하나라도 해당
  - 같은 입력을 서로 다른 관점에서 평가 (보안·성능·a11y·타입 등)
  - 생성과 검증이 분리되어야 함 (Producer-Reviewer)
  - 단계별 역할이 명확히 다름 (Analyst → Builder → QA)
  - 동적 경로 결정이 필요 (Supervisor)

- **단일 자산**: 한 역할이 처음부터 끝까지 담당해도 OK.

### (c) "모르겠음" 힌트

> *"아래 질문에 답해주세요: 이 작업에서 '서로 다른 사람이 검토해야 한다'는 느낌이 드는 부분이 있나요?*
> *- 있다 (생성과 검증이 같은 관점에서 나오면 오버핏 위험, 둘 이상의 축 독립 평가 필요) → 팀*
> *- 없다 (하나의 일관된 기준으로 충분) → 단일 자산"*

### (a) 팀 처리

Q3 skip. Phase 2에서 `harness-factory`로 위임.

전달 컨텍스트:
- 요구사항 한 문장 → harness-factory Phase 0의 도메인 설명
- Q2 답변 근거 → harness-factory Q1의 태스크 성격 힌트

### (b) 단일 자산 처리

Q3로.

## Q3. 트리거 방식 (skill vs command)

### 목적

같은 단일 자산이라도 **자동 발동** vs **명시 호출만** 구분. 전자는 skill, 후자는 command.

### 답변 분류

- (a) **자동 트리거** — 대화 중 키워드·의도에 따라 떠야 함 (예: "리팩터링"을 말하면 자동)
- (b) **명시 호출만** — 사용자가 `/cfh-something`을 입력할 때만 (예: 팀 규약 체크 커맨드)
- (c) **둘 다** — 자동 트리거 + 명시 호출 모두

### (c) 처리

**skill로 분류 + 동반 커맨드 생성**을 권장.

> *"skill은 자동 트리거를, 짝 커맨드(`/cfh-<name>`)는 명시 호출을 모두 커버할 수 있습니다. skill로 생성하면서 짝 커맨드를 동시에 만들까요? (y/n)"*

y → skill-author로 위임 + "짝 커맨드 포함" 플래그 전달. skill-author Phase 4(Write)에서 커맨드 파일도 함께 생성.

### (a) skill 처리

Phase 2에서 skill-author로 위임. Q1~Q2 답변 컨텍스트 전달.

### (b) command 처리

Phase 2에서 인라인 인터뷰 (3 질문). skill-author는 불필요.

## 단독 agent 분류 경로

Q1~Q3 어느 질문에서도 "agent"를 직접 묻지 않는 이유: **단독 agent 생성은 드문 케이스**이며, 대부분은 팀의 일부 또는 기존 skill에서 위임받는 구조이기 때문입니다.

다만 사용자가 명시적으로 "agent를 만들고 싶다"고 할 경우 또는 분류 결과가 명확히 agent인 경우 (예: "기존 스킬의 X 단계만 분리된 agent로 빼고 싶어") 아래 분기 질문을 추가:

```
단독 agent를 만드는 것은 아래 경우에만 권장됩니다:
(a) 기존 skill의 내부에서 위임받을 전문 역할을 분리 → `cfh new agent <name> --project` 스캐폴드 + 부모 skill 수정
(b) 새 팀의 첫 에이전트 → harness-factory 위임 권장 (나머지 에이전트도 한 번에 설계)
(c) 그 외 (독립 실행용) → 실제로는 command가 맞을 가능성. Q3 재분류.

어느 쪽인가요?
```

## 경계 케이스

### Case A. "매크로" 같은 것

사용자: "자주 쓰는 git 명령어 조합을 자동화하고 싶어"

- Q1: 반복 (a)
- Q2: 단일 (b)
- Q3: 명시 호출만 (b)
- → **command**

### Case B. 프로젝트 규약 자동 적용

사용자: "우리 팀은 에러 핸들링을 이렇게 하는데, Claude가 자동으로 알면 좋겠어"

- Q1: 반복 (a)
- Q2: 단일 (b) — 규약 주입은 한 관점
- Q3: 자동 트리거 (a)
- → **skill**

### Case C. PR 다축 리뷰

사용자: "PR 리뷰를 보안·성능·타입 다 봐줬으면"

- Q1: 반복 (a)
- Q2: 팀 (a) — 서로 다른 축 독립 평가
- → **team** (harness-factory의 Expert Pool 패턴 가능성 높음)

### Case D. 애매한 TDD 보조

사용자: "TDD할 때 자꾸 구현을 먼저 보게 되는데, Claude가 막아줬으면"

- Q1: 반복 (a)
- Q2: (c) 모르겠음 → 힌트 제시 → "생성자·검증자 분리되면 좋겠어" 답변
- → **team** (Producer-Reviewer)

단, Q2에서 "하나의 일관된 기준으로 충분"이면:
- Q3: 자동 트리거 (a)
- → **skill** (tdd-first 확장 가능성 있으니 중복 확인)

### Case E. 이미 tdd-first가 있는데 비슷한 것 만들고 싶어함

Pre-scan에서 `tdd-first` 매칭 → (a) 확장·편집 권장. asset-factory는 "skill-author에 '기존 스킬 편집' 모드로 위임" 하는 대안도 고려.
