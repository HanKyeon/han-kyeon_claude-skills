이 커맨드는 **증거 주도** 조사 워크플로입니다. 목표(goal)가 아닌 **증상(symptom)**에서 시작해 체계적으로 근본 원인을 찾습니다. 결과물은 "원인 확정 + 수정 계획"이며, 실제 수정은 `/cfh-plan`·`/cfh-tdd`·`/cfh-refactor` 등으로 위임합니다.

**호출 경로**: 이 커맨드로 **명시 호출**하거나, `debug-investigator` 스킬이 **자동 트리거** (증상·에러·stack trace·장애 언급 시). 둘 다 같은 5 Phase 워크플로 진입.

**핵심 원칙**:
- **증거부터, 가설은 그 뒤.** 가설을 먼저 세우고 증거를 맞추는 방식 금지 (확증 편향).
- **최소 3개 가설.** 1개만 세우면 그것만 보게 됨. 3개를 병렬 나열해 상대 비교.
- **가설 제거는 체계적으로.** Git bisect, 바이너리 검색, A/B 재현 등 방법론 명시.
- **근본 원인 입증 없이 수정 금지.** "이게 원인일 거야"로는 부족함. 재현 또는 명확한 증거 체인 필요.
- **긴급도 > 완전성.** 운영 장애면 hotfix 먼저, proper fix 나중. `/cfh-plan`에서 정식 수정.
- **조사 내용 기록.** 시간 낭비 않게 `DEBUG-LOG.md` 또는 Issue에 발자국 남김.

## Phase 0 — Evidence Gathering (증거 수집)

**원칙**: 이 단계 **끝나기 전 코드 수정 금지**. 증거가 얕으면 가설이 얕아짐.

### 0-1. 이슈 유형 분류

사용자에게 질문:

```
어떤 종류의 이슈입니까?

(a) 기능 버그 — 특정 시나리오에서 기대와 다른 결과
(b) 장애 / 인시던트 — 프로덕션 실패·파셜 다운·알람
(c) 성능 이슈 — 느림·메모리·렌더 지연·쿼리 지연
(d) 회귀 — 이전에는 작동했는데 특정 시점 후 안 됨
(e) 복합 — 둘 이상 섞임 (예: 회귀로 인한 성능 저하)
```

유형별로 Phase 0-2 증거 질문 세트가 다릅니다 (아래).

### 0-2. 유형별 증거 질문

#### (a) 기능 버그

```
Q1. 증상: 무엇이 **기대와 다른가**? (구체적 시나리오로 — 클릭 → 결과)
Q2. 재현 단계: 1, 2, 3, ... 최소 정보로 재현되는 절차
Q3. 재현 빈도: 항상 / 가끔 (빈도·조건) / 한 번만
Q4. 환경: 브라우저·OS·사용자 role·데이터 조건·feature flag
Q5. 에러 메시지·스택 트레이스·로그: 있는 그대로 붙여넣기
Q6. 영향 범위: 전 사용자·일부 segment·개발 환경만
```

#### (b) 장애 / 인시던트

```
Q1. 증상: 무엇이 멈췄거나 잘못되고 있나? 알람·사용자 제보?
Q2. 시작 시점: 언제부터? 모니터링 그래프에서 스파이크 시간
Q3. 영향 범위: 전체 / 특정 지역·서비스 / 특정 사용자 segment
Q4. 심각도: 완전 중단 / 부분 기능 실패 / 성능 저하
Q5. 최근 변경: 배포·인프라 변경·3rd party 변화·traffic 급증?
Q6. 즉시 rollback 가능? yes/no — yes면 "불 먼저 끌까요?" 분기
Q7. 로그·메트릭·알람: 수집된 원시 데이터
```

**긴급 분기**: Q6 = yes & 심각도 높음 → **즉시 rollback 권장**. 조사는 그 후 진행.

#### (c) 성능 이슈

```
Q1. 어떤 지표가 악화됐나? (응답시간·LCP·INP·메모리·CPU·쿼리시간)
Q2. 얼마나? 기준치 대비 몇 배·몇 ms 증가
Q3. 언제부터? (특정 시점 / 서서히)
Q4. 어떤 경로·작업에서? 특정 API·페이지·쿼리
Q5. 측정 방법: Profiler·Lighthouse·APM·로그 집계
Q6. 환경 차이: 로컬에서도 재현 / 프로덕션만
```

#### (d) 회귀

```
Q1. 이전에는 **정확히 어떻게 작동했나?** 기준선 확인
Q2. 마지막으로 작동 확인한 **버전·날짜·commit**
Q3. 첫 고장 확인한 **버전·날짜·commit**
Q4. 그 사이에 있었던 변경 후보 (배포·의존성 업데이트·config·DB migration)
Q5. 재현 단계·환경 (버그 Q2~Q4와 동일)
```

회귀는 Phase 3에서 **git bisect**가 핵심 도구.

#### (e) 복합

각 유형의 질문을 교집합으로 수행. Claude가 유형별 가중치를 판단 후 적절한 질문 병합.

## Phase 1 — Context Scan (컨텍스트 스캔)

**증거를 바탕으로 scoped 스캔**. broad scan 금지 — Phase 0이 시점·파일을 좁혔으니 거기 맞춰서.

### 수집 대상

- **Git 이력** (최근 변경):
  - 유형 (d) 회귀: `git log --since "<Q2 날짜>" --until "<Q3 날짜>"` 혹은 `git log <last-good>..<first-bad>`
  - 기타: `git log --oneline -20 <대상 파일·디렉터리>` + 관련 파일 `git blame`
- **관련 PR·이슈** (gh CLI 있으면):
  - 최근 머지된 PR 중 대상 영역 언급
  - 유사 과거 이슈·버그 (제목·본문 키워드 매칭)
- **스택 트레이스 파일**:
  - 에러 메시지에 등장한 파일·라인 중심으로 주변 코드 스캔
- **의존성 변경**:
  - 유형 (c)/(d)에서 중요 — `package-lock.json`·`pnpm-lock.yaml` diff 확인
- **CI/배포 이력**:
  - `.github/workflows/`·배포 로그 — (b) 장애에서 중요

```

     - [inferred] 확인 비용 가장 낮음 — <도구·시간 추정 1줄>

다른 옵션:
   - 가설 <M>부터 — <조건: 예 "프로덕션 환경 접근이 어려울 때 dev 재현 우선">
```

이유에 인용할 신호 (택 1~2):
- 시점: "배포 직후"·"특정 commit 이후" → bisect 가설 우선
- 환경 격차: "프로덕션만"·"특정 브라우저만" → 환경 가설 우선
- 재현 난이도: 100% 재현 → 빠른 확인 / 산발적 → 로그 기반 가설 우선

상세 추천 패턴: `commands/references/recommendation-pattern.md`.

### 유형별 가설 체크리스트 (놓치기 쉬운 축)

**(a) 기능 버그**:
- 조건 분기 누락 (null·empty·경계)
- 상태 관리 race (stale closure·업데이트 순서)
- 이벤트 전파 (preventDefault·stopPropagation)
- API 계약 불일치 (스키마 drift)

**(b) 장애**:
- 배포 실패·환경변수 누락
- DB·캐시 장애 (커넥션 풀·락)
- 3rd party 장애 (Stripe·Auth0 등)
- Traffic spike → rate limit·timeout
- Config 변경 (feature flag·권한)

**(c) 성능**:
- N+1 쿼리·missing index
- 렌더 원인 (객체 재생성·불필요 re-render)
- 번들 사이즈 증가 (lodash 전체 import 등)
- 비동기 순서 (waterfall vs parallel)
- 메모리 누수 (리스너·타이머)

**(d) 회귀**:
- 특정 커밋 (Phase 3 bisect)
- 의존성 메이저 업데이트
- Config·feature flag 변경
- DB migration
- 브라우저·런타임 버전 변화

## Phase 3 — Verification (가설 검증)

**가설 하나씩 체계적으로 검증·제거**. 근거 없이 "이게 맞다"고 수렴 금지.

### 유형별 검증 도구

#### (a) 기능 버그
- **재현 테스트 작성**: Phase 0 Q2 재현 단계를 자동화 테스트로 옮김
- 테스트 작성 중 가설이 확정되면 곧바로 Phase 4

#### (b) 장애
- **로그 집계·grep**: 에러 패턴·시작 시점·영향 범위 수치화
- **rollback A/B**: 직전 배포로 되돌려 재현 안 되면 신규 배포가 원인 확정
- **3rd party status**: 외부 의존성 상태 페이지·health check

#### (c) 성능
- **Profiler·Flamegraph**: Chrome DevTools·React Profiler·`pprof`·EXPLAIN ANALYZE
- **기준치 측정**: 동일 경로 N회 수행, p50·p95
- **before/after 비교**: 최근 변경 이전 버전에서 측정

#### (d) 회귀
- **Git bisect**: 마지막 good → 첫 bad 사이 이분법
  ```bash
  git bisect start
  git bisect bad <first-bad>
  git bisect good <last-good>
  # 각 단계: 재현 수행 → git bisect good|bad
  ```
- **의존성 diff**: `npm diff <pkg>@<old> <pkg>@<new>` 또는 lock 파일 비교

```
  → 큰 리팩터 필요하면 /cfh-refactor
```

### 조사 로그 저장

Phase 0~3의 증거·가설·검증 과정을 **DEBUG-LOG.md** 또는 해당 Issue에 남겨 두어야 후속 작업자가 동일 작업을 반복하지 않음.

```
DEBUG-LOG.md 또는 Issue #<N>에 조사 내역 기록 안내:
- 증거 카드 (Phase 0)
- Context Scan 결과 (Phase 1)
- 가설 N개 + 검증 결과 (Phase 2·3)
- 확정된 원인 + 수정 계획 (Phase 4)
```

## Phase 복귀 규칙

Phase 3 이후에 초반 단계로 돌아가야 할 때:
- **"증거 재수집"** → Phase 0 복귀 (재현 안 되거나 환경 정보 부족할 때)
- **"context 재스캔"** → Phase 1 복귀 (시점·범위 재지정)
- **"가설 재구성"** → Phase 2 복귀 (기존 가설 전부 거짓일 때)
- **"검증 방법 변경"** → Phase 3 재시도 (도구·접근 바꿈)

git은 조사 과정을 되돌릴 수 있게 해주지 않으므로, **Phase별 카드를 각각 명시적으로 출력·보존**하는 것이 되돌림의 근거가 됩니다.
