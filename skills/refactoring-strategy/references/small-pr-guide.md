# Small PR Guide

## 왜 작은 PR인가

- **리뷰 가능성**: 500줄 넘는 PR은 사람이 제대로 리뷰하지 못한다. 연구에 따르면 400줄 이상에서 결함 발견율이 급락한다.
- **롤백 가능성**: 문제가 발견됐을 때 되돌릴 범위가 좁다.
- **행동 보존 검증**: 변경이 좁을수록 "정말 동작이 같은가"를 확인하기 쉽다.

## 크기 가이드라인

| PR 규모 | 변경 라인 | 파일 수 | 리뷰 난이도 |
|---|---|---|---|
| Tiny | < 50 | 1~2 | 즉시 |
| Small | 50~200 | 2~5 | 쉬움 |
| Medium | 200~500 | 5~10 | 집중 필요 |
| Large | 500~1000 | 10~20 | 쪼갤 것 |
| Huge | 1000+ | 20+ | **반드시 쪼갤 것** |

리팩터링은 **Small 이하**를 목표로 한다.

## 쪼개는 전략

### 1. 수직 분할 (Vertical Slice)
한 기능을 완전히 하나의 PR로. 예: "환자 테이블의 혈압 셀 리팩터링" — 컴포넌트 + 타입 + 테스트 + 관련 훅을 하나로.

### 2. 수평 분할 (Horizontal Slice)
공통 패턴을 여러 파일에 적용. 예: "모든 폼의 RHFTextField를 새 API로 마이그레이션"은 파일 5개 단위로 쪼갬.

### 3. Scaffolding PR
새 구조를 **비어 있는 상태**로 먼저 머지. 그 다음 PR에서 내용 채움. 리뷰어가 구조를 먼저 합의할 수 있다.

### 4. Adapter PR
기존 코드와 신 코드가 공존하는 중간 상태를 만드는 PR. 그 다음 PR에서 소비자를 하나씩 이전.

## 커밋 메시지 컨벤션

```
refactor: <무엇을 어떻게>

Why: <이유>
What NOT changed: <명시적으로 안 바꾼 것>
Risk: <놓칠 수 있는 영향>
```

예:
```
refactor: extract BloodPressureCell from PatientTable

Why: PatientTable이 800라인으로 커져서 셀 단위 테스트가 어려움
What NOT changed: API 호출, 렌더링 결과, props 인터페이스
Risk: className 적용 순서가 미묘하게 달라질 수 있음 (smoke test로 확인)
```

## 반드시 피할 것

- ❌ "이왕 여기 온 김에 이것도 고침" 커밋
- ❌ "refactor + feat" 혼합 커밋
- ❌ 빈 import 정리만 10개 파일 건드리는 커밋 (별도 PR로)
- ❌ 자동 포매터 실행 결과를 다른 변경에 섞음 (별도 커밋)

## Pre-PR 체크리스트

- [ ] 이 변경은 **동작을 바꾸지 않는가**?
- [ ] 커밋 메시지가 **Why**를 설명하는가?
- [ ] **Scope 외** 파일을 건드렸는가? → 다시 꺼낼 것
- [ ] 테스트가 모두 GREEN인가?
- [ ] Smoke test로 주요 화면 확인했는가?
