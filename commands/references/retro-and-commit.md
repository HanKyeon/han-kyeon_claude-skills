# Retro + Commit 블록 (공유 템플릿)

작업이 완료된 시점에 커맨드 종료 보고에 **반드시** 포함해야 하는 두 블록의 표준 형식입니다.
`/cfh-plan`·`/cfh-tdd`·`/cfh-refactor`·`/cfh-debug`·`/cfh-review`가 동일한 형식을 사용합니다.

## 원칙

1. **생략 금지.** 작업이 완료(또는 단계 완료)됐다면 두 블록을 **모두** 출력. 사소한 작업이어도 1~2줄로 압축은 OK, 누락은 안 됨.
2. **자동 commit·자동 저장 금지.** 두 블록은 **제안**일 뿐. 사용자가 명시 yes 해야 진행.
3. **재사용 우선.** Q1·Q2·이전 보고에 이미 있는 정보는 재인용 — 새로 묻지 않음.

---

## 블록 1 — 🔄 Retro (작업 회고)

3섹션 고정. 각 섹션은 **bullet 1~3개**. 비어있어도 섹션은 출력 (빈 경우 "해당 없음").

```
🔄 Retro

효과 있었음 (계속할 것):
  - <접근법·도구·순서 중 잘 작동한 것 — 다음에도 반복할 만한 것>

실패·삽질 (반복 금지):
  - <헛수고였던 시도·잘못된 가정·우회 — 회피 신호와 함께>
  - 해당 없음 (실패 없음)

다음엔 바꿀 것:
  - <개선 아이디어·후속 과제 — 이번 작업에선 반영 안 함>
  - 해당 없음

저장: /cfh-retro로 영구 기록 가능 (선택)
```

### 섹션별 작성 가이드

**효과 있었음**:
- 단순 "잘 됐다"가 아니라 *왜 잘 됐는지*. 예: "Q1에서 경로를 좁혀서 broad scan 회피"
- 도구·순서·휴리스틱 중심으로

**실패·삽질**:
- "처음에 X를 시도했는데 Y 이유로 폐기" 형식
- 사용자가 거부했던 접근법도 포함 — 다음 세션에 동일 제안 반복 방지
- 진짜 없으면 "해당 없음"

**다음엔 바꿀 것**:
- 이번에 반영하지 못한 후속 — 별도 PR·Issue 후보
- "Q3 out-of-scope"였던 항목과 구분 (그쪽은 완료 보고의 별도 섹션)

## 블록 2 — 📝 제안 커밋

작업이 **코드 수정을 동반**한 경우에만 출력. 탐색·분석·리뷰만 한 경우엔 생략하고 한 줄로 "코드 수정 없음 — 커밋 제안 생략" 표시.

```
📝 제안 커밋

메시지 초안:
  <subject — 컨벤션에 맞춰 1줄, 50자 내외>

  <body — 왜 변경했는지 1~3문장. 무엇을 했는지는 diff가 말해줌>

스테이지 범위:
  + <추가 파일>
  ~ <수정 파일>
  - <삭제 파일>
  (제외: .env·credentials.* 등 비밀파일은 자동 제외)

분할 추천:
  - 단일 커밋 권장 — 한 관심사
  또는
  - 2개로 분할 권장:
    1) <범위 A> — "<msg-A>"
    2) <범위 B> — "<msg-B>"
    근거: <다른 top-level 디렉터리 / 다른 관심사 / 리팩터+기능 혼재>

이대로 진행하시겠습니까?
- (yes) git add <범위> + git commit
- (edit-msg) 메시지 수정
- (split-differently) 분할 재제안
- (no-commit) 커밋하지 않음 (사용자가 직접 처리)
```

### 메시지 컨벤션 추정 절차

1. `git log -10 --pretty=format:"%s"` 실행 → 최근 10개 subject
2. 다음 패턴 중 가장 빈번한 것 채택:
   - `feat: <…>` / `fix: <…>` / `refactor: <…>` / `docs: <…>` (Conventional Commits)
   - `[<scope>] <…>` (Square-bracket scope)
   - `<verb> <…>` (자유형)
3. 톤 (영어/한국어, 명령형/과거형) 일치
4. 추정 결과를 메시지 위에 한 줄로 공개:
   ```
   (컨벤션 추정: Conventional Commits, 영어, 명령형)
   ```

### 비밀파일 자동 제외 규칙

다음 패턴은 **반드시 스테이지 범위에서 제외** (사용자가 명시적으로 포함 요청해도 한 번 더 확인):
- `.env`, `.env.*`
- `*credentials*`, `*secret*`, `*.key`, `*.pem`
- `id_rsa`, `id_ed25519` 등 개인키
- `~/.aws/`, `~/.ssh/` 경로 내 파일

### 분할 휴리스틱

다음 신호 중 **2개 이상** 충족 시 분할 추천:
- 다른 top-level 디렉터리 (`src/` vs `tests/`는 같은 관심사로 봄, `src/feat-A/` vs `src/feat-B/`는 분할)
- 리팩터 + 기능 추가 혼재 (커밋 메시지가 "feat + refactor" 처럼 두 종류)
- 무관한 의존성 추가 (`package.json`만 + 별도 기능 코드)
- 5개 이상 파일 + 2개 이상 다른 모듈

단일 신호만으론 분할 강요 안 함. 사용자가 "한 번에 가자" 거부 가능.

## 출력 위치

각 커맨드의 **완료 보고 블록 안**, 다음 순서로:

```
✅ 작업 완료 보고
  - 목표 달성: …
  - 생성·수정 파일: …
  - 검증 수행: …

🔄 Retro
  …

📝 제안 커밋
  …

다음 단계:
  - …
```

## 위임 시

`/cfh-plan`이 `/cfh-tdd` 등에 위임한 경우, 위임받은 커맨드가 **자체 완료 보고에 두 블록 출력**.
`/cfh-plan`은 위임 후 자기 보고에서 두 블록을 다시 출력하지 않음 (중복 회피).
