<s>
**🧪 experimental (0.24.0 시범)** — AskUserQuestion 도구로 *선택지 기반 결정*을 받는 실험 커맨드. 가치 검증 후 `/cfh-plan` 등에 확산하거나 제거 결정. 기존 자산에 영향 없음.

**🔀 잘못 진입하셨다면**:
- 복잡한 다단계 작업을 시작 → `/cfh-plan`
- 결정 트리를 깊이 파기 → `/cfh-grill`
- 옵션이 5개 이상이거나 자유 서술이 더 맞는 결정 → 그냥 자연어로 대화

이 커맨드는 **가벼운 단일 결정**을 *구조화 대화*로 진행합니다. 추천 + 이유 + 옵션을 **텍스트로 모두 출력**한 뒤, `AskUserQuestion` 도구로 선택을 받습니다. 자연어 자유 답변보다 *약간 제약이 걸린* 형태 — 선택지가 명시되고, 추천이 첫 번째로 강조됩니다.

**핵심 원칙**:
- **설명은 텍스트로 풀 출력.** 추천 이유(`[verified]/[inferred]/[guessed]`)는 절대 압축하지 않음 — 도구의 짧은 description에 욱여넣지 말 것.
- **답변만 도구로 수집.** `AskUserQuestion`은 *선택 UI*로만 사용.
- **작은 범위.** 옵션 ≤ 4개인 단일 결정만. 초과하면 자연어/`cfh-plan`으로 안내.
- **fallback 필수.** 도구를 쓸 수 없는 환경(서브에이전트·일부 클라이언트)에서는 텍스트 번호 선택으로 자동 전환.
</s>

<invocation>
구조화 결정 대화를 시작합니다.

**인자**: `$ARGUMENTS` — 결정할 내용 한 문장(선택). 예: "state를 zustand에 둘지 로컬에 둘지", "에러 UX를 toast로 할지 inline으로 할지".

- 비어있으면 "어떤 결정을 도와드릴까요?"로 질문.
- 값이 있으면 그대로 결정 대상.
</invocation>

<workflow>

## Step 1 — 결정 대상 수집

`$ARGUMENTS` 또는 재질문 답변을 결정 대상으로 확정. 한 문장으로 재진술해 사용자 의도를 맞춤.

## Step 2 — 옵션 수 사전 점검 (제약 가드)

이 커맨드는 **옵션 2~4개**만 다룹니다 (`AskUserQuestion` 하드 제한).

- 합리적 선택지가 **5개 이상**이거나 *자유 서술*이 본질인 결정이면 → 진행하지 말고 안내:
  ```
  이 결정은 선택지가 많아(또는 서술형이라) /cfh-ask에 잘 안 맞습니다.
  자연어로 대화하거나 /cfh-plan으로 진행하시길 권합니다.
  ```
- 2~4개로 수렴 가능하면 Step 3.

## Step 3 — 컨텍스트 확인 (추천 근거, 선택)

추천에 근거가 필요하면 **사실만** 가볍게 확인 (읽기 전용):
- 코드 현황(`Grep`/`Glob`/`Read`) — 추천 이유의 `[verified]` 근거로만
- `CLAUDE.md` 규약 — 프로젝트 컨벤션 신호

**원칙**: 코드는 *현재 상태*(이유 근거), 결정은 *사용자 의도*. 코드 현황으로 결정을 자동 확정하지 말 것 (→ `commands/references/recommendation-pattern.md`).

## Step 4 — 설명 텍스트 풀 출력 (압축 금지)

`AskUserQuestion` 호출 **전에**, 추천+이유+옵션을 텍스트로 모두 출력 (→ `commands/references/recommendation-pattern.md` · `confidence-tagging.md`):

```
📌 추천: <옵션 A>
   이유:
     - [verified] <코드·발화 인용>
     - [inferred] <합리적 추론>
     - [guessed] <약한 신호 — 확인 권장>

다른 옵션:
   - <B> — <어떤 조건일 때 적합>
   - <C> — <조건>
```

이 단락이 **결정의 실질 정보**입니다. 아래 도구는 *선택 수단*일 뿐.

## Step 5 — AskUserQuestion으로 선택 수집

위 설명을 출력한 뒤 `AskUserQuestion` 도구를 호출:

- `question`: 결정 한 문장 (위에서 설명했으므로 짧게)
- `header`: ≤ 12자 라벨 (예: "State 위치")
- `options`: 2~4개
  - **첫 옵션 = 추천**, label 끝에 `(추천)` 표기
  - `description`: *짧게* (한 줄). 상세 이유는 Step 4 텍스트에 이미 있음
  - (선택) `preview`: 옵션별 트레이드오프를 비교하고 싶을 때만
- `multiSelect`: 상호배타가 아닌 결정이면 `true` (예: "어떤 점검을 켤까")
- **"Other"는 자동 제공** — 사용자가 추가 조건을 달거나(`"B로 하되 TTL 60초"`) 제3안을 낼 수 있음

### Fallback — 도구를 쓸 수 없을 때

서브에이전트 컨텍스트(`AskUserQuestion`은 Agent tool 스폰 시 불가)이거나 도구가 없으면, **Step 4 설명은 그대로 두고** 답변만 텍스트로:
```
선택: 1 (추천) / 2 / 3 / 기타 <자유 입력>
```

## Step 6 — 선택 반영

받은 선택(또는 Other 입력)을 확정하고 다음으로:
- 결정이 *작업으로 이어지면* → 적절한 커맨드 안내 (`/cfh-tdd`·`/cfh-refactor` 등) 또는 직접 수행
- 결정이 *정보 확정이면* → 한 줄 확정 요약
- "Other"로 새 조건이 들어오면 → 그 조건을 반영해 재확인 (필요 시 Step 4~5 1회 반복)

## Step 7 — 워크플로 중간 호출 처리 (조건부)

cfh-ask가 *다른 워크플로 진행 중*에 호출됐는지 **컨텍스트로 판단** (slash command는 컨텍스트를 공유하므로 직전 대화에 cfh-plan/grill 등의 진행 상태가 남아 있음):

- **다른 워크플로 감지** (cfh-plan Phase 진행 중 · cfh-grill 트리 walk 중 등):
  - 확정 답을 그 워크플로의 *해당 Phase/결정 답변*으로 컨텍스트에 남김
  - 복귀 안내 출력: `↩ <원래 워크플로>로 돌아갑니다 — 이 답(<선택>)은 해당 Phase 답변으로 반영됩니다.`
  - 원래 워크플로의 다음 단계로 자연스럽게 이어감
  - **별도 "수집·전달" 배관 없음** — 같은 대화라 확정 답이 원래 워크플로의 LLM에게 그대로 보임. cfh-plan이면 Phase 2.5 Final Intent Confirm이 이 답까지 종합.
- **독립 호출** (다른 워크플로 컨텍스트 없음):
  - 이 Step skip — Step 6에서 일반 종료. 엉뚱한 "복귀" 안내 금지.

**자가검증**: 워크플로 감지가 불확실하면 *복귀 안내를 강제하지 말 것*. 독립 호출에 "원래 흐름 복귀" 같은 잘못된 메시지를 내지 않도록 — 명확한 진행 워크플로 신호가 있을 때만 복귀 처리.

</workflow>

<output_format>

Step 4 설명 카드 → Step 5 도구 호출 순서. 도구 결과(선택)를 받은 뒤 한 줄 확정:

```
✅ 결정: <선택된 옵션> <(+ Other 조건 있으면)>
   다음: <이어질 작업·커맨드 또는 "정보 확정">
```

</output_format>

<constraints>
- **옵션 2~4개 전용.** 5개+ 또는 서술형 결정은 거절하고 자연어/`cfh-plan` 안내 (Step 2).
- **설명 압축 금지.** 추천 이유는 Step 4 텍스트에 풀 출력. `AskUserQuestion` description은 한 줄 요약만.
- **결정 권한은 사용자.** 코드 현황은 이유 근거로만 인용, 자동 확정 금지.
- **fallback 필수.** 서브에이전트·도구 부재 환경에서 텍스트 번호 선택으로 전환 (Step 5 Fallback).
- **experimental.** 이 커맨드는 시범 — 사용 후 가치 없으면 제거 대상. `/cfh-feedback` 으로 사용 경험을 남기면 확산/제거 판단에 활용. 검증되면 cfh-plan·cfh-grill에 *조건부 구조화 선택*으로 내장 예정 (발동 기준 = 이 실험에서 도출).
- **워크플로 중간 호출 시** Step 7 조건부 복귀, **독립 호출 시** Step 6 일반 종료 — 두 경로 모두 매끄럽게.
- 한국어 설명, 기술 용어 영어 병기.
</constraints>
