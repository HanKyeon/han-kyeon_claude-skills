# Interview Flow — Phase 1 상세

기본 5 질문 → (옵션) Deep-dive 7 축 → Sanity check → 요약 카드 순. Deep-dive는 **사용자 옵트인**이며, 승인 시에도 Q1~Q5 답변에 따라 **관련 축만 조건부로** 꺼냅니다.

## 원칙

1. **기본 5개 먼저 완료.** Deep-dive에 들어가기 전에 기본 답 확보.
2. **Deep-dive는 선택.** 요약 카드 직전에 "더 깊이 설계하시겠습니까? (y/n)" 제시. n이면 바로 Sanity check.
3. **관련 축만.** y라고 해도 7개 전부 묻지 않음. Q1~Q5 답변으로 **필요한 축만** 선택.
4. **가정 금지.** 답이 모호하면 (a)/(b)/(c) 옵션으로 재질문.

---

## 기본 5 질문

### Q1. 태스크 성격

> *"이 팀이 다룰 전형적 태스크를 1~2 문장으로 설명해주세요. 그리고 아래 중 가장 가까운 것을 고르세요."*

- (a) **선형 변환** — 정해진 단계로 입력을 결과물로 변환 (예: 요구사항 → 스펙 → 코드 → 테스트)
- (b) **병렬 분해** — 독립된 부분 문제를 동시에 처리 후 결과 병합
- (c) **다축 평가** — 같은 입력을 여러 전문 관점에서 각각 평가 (PR 리뷰 등)
- (d) **검증 중심** — 생성과 검증이 분리되어야 함 (오버핏 방지)
- (e) **동적 분기** — 실행 중에 다음 단계가 결정됨
- (f) **계층 분해** — 큰 이슈가 하위 팀 2개 이상으로 자연스럽게 나뉨

→ 첫 패턴 후보: (a) Pipeline, (b) Fan-out/Fan-in, (c) Expert Pool, (d) Producer-Reviewer, (e) Supervisor, (f) Hierarchical.

### Q2. 입력·출력

> *"한 번의 호출에 들어오는 입력은 무엇이고, 최종 산출물은 무엇입니까?"*

예시 답변:
- 입력: "GitHub diff + 프로젝트 CLAUDE.md", 출력: "REVIEW.md 리포트"
- 입력: "기능 요구사항 1문단", 출력: "구현 파일 + 테스트 파일"
- 입력: "마이그레이션 대상 폴더", 출력: "PR 시리즈"

### 답변 해석
- 입력이 구조화(예: diff) → 분할 용이 → Fan-out 가능.
- 출력이 단일 리포트 → Aggregator/Merger 필요.
- 출력이 코드 + 테스트 → Producer-Reviewer 고려.

### Q3. 전문성 축

> *"이 태스크에서 독립적으로 평가되어야 할 축은 무엇입니까? (예: 보안, 성능, 접근성, 타입, 도메인)"*

- 1 축 → 단일 에이전트로 충분. 패턴 불필요.
- 2~3 축 → Pipeline 또는 Expert Pool.
- 4~6 축 → Expert Pool.
- 축이 겹침 → 병합 검토.

### Q4. 실패 비용

> *"이 팀의 산출물이 틀리면 되돌리는 비용은? (a) 즉시 롤백 쉬움 (b) 커밋 후 발견 시 수정 비용 큼 (c) 운영 장애·데이터 손실 위험"*

- (a) → 일반 패턴 OK.
- (b) → Producer-Reviewer 권장.
- (c) → Producer-Reviewer 강제 + 사람 승인 관문 추가.

### Q5. 규모

> *"몇 개의 서로 다른 전문 역할이 필요합니까?"*

- 2~3 → Pipeline / Producer-Reviewer.
- 3~6 → Expert Pool / Supervisor.
- 6+ → Hierarchical Delegation 고려.

## Deep-dive 옵트인 게이트

기본 5 질문 완료 후 사용자에게:

```
기본 인터뷰가 끝났습니다.

더 깊이 설계하시겠습니까?
- (y) 추가 축 질문 진행 — 통신·상태성·실패 처리 등 Q1~Q5 답변과 관련된 것만 선별
- (n) 이대로 Sanity check와 패턴 추천으로 진행

추천: <자동 추천 — 아래 규칙 기반>
```

### 자동 추천 규칙

- Q4 = (c) 또는 Q5 = 6+ → **(y) 권장**
- Q1 = (e) Supervisor 또는 (f) Hierarchical → **(y) 권장**
- Q5 = 2~3 AND Q4 = (a) → **(n) 권장** (단순 팀은 기본만으로 충분)
- 그 외 → 사용자 선택

## Deep-dive 7 축 — 조건부 노출

사용자가 (y) 선택 시 아래 7개 축 중 **기본 답변과 관련된 축만** 질문합니다. 무관한 축은 건너뜁니다.

### D1. 통신 대역폭

> *"에이전트 간에 전달되는 페이로드는 얼마나 큽니까?"*

- (a) 한 문장 스펙 / 짧은 판정 (작음)
- (b) 수 KB 리포트·diff (중간)
- (c) 수십 KB 이상 원본 파일 다수 (큼)

**활성 조건**: Q1 = (a)/(b)/(f) OR Q5 = 4+
**영향**:
- (c) 큰 경우 → 각 에이전트에 `Read` 권한 필수, 페이로드 분할·요약 전담 오케스트레이션 필요.
- (a) 작은 경우 → 에이전트 경량화 가능.

### D2. 상태성

> *"에이전트가 호출 사이에 상태를 기억해야 합니까, 매번 독립입니까?"*

- (a) 완전 독립 (stateless)
- (b) 세션 내 기억 (orchestrator가 전달)
- (c) 영속 상태 (파일/DB)

**활성 조건**: Q1 = (e) Supervisor OR (f) Hierarchical
**영향**:
- (c)는 stateless 에이전트 전제가 깨지므로 오케스트레이터가 상태 주입 전담.

### D3. 인간 개입 지점

> *"어느 단계에 사람 확인이 필요합니까?"*

- (a) 완전 자동
- (b) 단계 경계마다 승인
- (c) 특정 단계에만 승인 (명시 필요)

**활성 조건**: Q4 = (b) OR (c)
**영향**:
- (b)/(c)는 pause gate 설계 필요. 에이전트 `description`에 "사용자 승인 대기" 명시.

### D4. 반복 한계

> *"에이전트가 같은 단계를 반복할 수 있는 설계라면, 최대 반복 수는?"*

- (a) 반복 없음 (한 번 호출)
- (b) 최대 N회 (지정)
- (c) 수렴 조건 (구체 조건)

**활성 조건**: Q1 = (e) Supervisor OR Deep-dive에서 "수정 피드백 루프" 답변
**영향**:
- 무한 루프 방지 — 오케스트레이터에 반복 카운터 필수.

### D5. 권한·도구

> *"에이전트별로 필요한 최소 도구는? 파일 쓰기 권한이 있어야 할 에이전트는 누구입니까?"*

- 기본: 모든 에이전트 `Read, Grep, Glob`
- 추가 필요: 에이전트별로 `Write` / `Edit` / `Bash` 중 어느 것?

**활성 조건**: 항상 (Deep-dive 진입 시 공통 질문)
**영향**:
- 최소 권한 원칙 적용. Producer만 Write, Reviewer는 Read 전용 식.

### D6. 실패 처리

> *"한 에이전트가 실패했을 때 어떻게 처리합니까?"*

- (a) 오케스트레이터가 롤백
- (b) 다음 에이전트로 skip
- (c) 대체 에이전트에 재위임
- (d) 사용자에게 에스컬레이션

**활성 조건**: Q4 = (b) OR (c)
**영향**:
- (a) 롤백 필요 → 각 에이전트가 변경분을 요약 반환하도록 output contract 강제.
- (c) 대체 → Supervisor 패턴으로 자동 전환 검토.

### D7. 관측성

> *"각 에이전트의 판정·산출물을 어디에 기록합니까?"*

- (a) 표준 출력만
- (b) 전용 로그 파일 (경로 명시)
- (c) CI 시스템 연동 (PR 코멘트 등)

**활성 조건**: Q4 = (c) OR Pre-scan에서 `.github/workflows/` 발견
**영향**:
- (b)/(c) → 에이전트 output contract에 "로그 경로/형식" 명시 필수.

## Deep-dive 실행 순서

```
1. Q1~Q5 기본 답변 완료
2. 옵트인 게이트 (y/n) + 자동 추천
3. n → Sanity check로 바로 이동
4. y →
   a. 활성 조건 만족 축만 필터링 (예: D2·D4는 Q1이 (e)일 때만)
   b. D5(권한)는 항상 포함
   c. 일반적으로 2~4개 축이 활성화됨. 7개 전부 묻는 일은 드묾.
5. 모든 응답 수집 후 Sanity check (`sanity-check.md`의 R1~R7)
```

## Phase 1 종료 — 요약 카드

기본 5 + Deep-dive(있으면) + Sanity check 완료 후:

```
=== 팀 설계 요약 (초안) ===
태스크 성격: <Q1>
입력 → 출력: <Q2>
전문성 축: <Q3>
실패 비용: <Q4>
규모: <Q5>

(Deep-dive 수행 시에만)
통신 대역폭: <D1>
상태성: <D2>
인간 개입: <D3>
반복 한계: <D4>
권한·도구: <D5>
실패 처리: <D6>
관측성: <D7>

Sanity check: ✅ 통과 (또는 재확인 완료)

추천 패턴: <이름> (대안: <다른 이름>)
이유: <3줄 이내 — Deep-dive 결과를 근거로 포함>

Phase 2(패턴 확정)로 진행해도 될까요?
```

사용자 승인 후에만 Phase 2로.
