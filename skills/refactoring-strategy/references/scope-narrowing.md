# Refactoring Scope Narrowing Protocol

리팩터링 작업 시 **코드를 건드리기 전에** 범위를 질문으로 좁히는 프로토콜입니다. Scope 폭발을 막는 가장 효과적인 장치입니다.

## 핵심 질문 8가지

작업 대상이 확정됐더라도 아래 질문을 **순서대로** 사용자에게 확인합니다. 답변이 불충분하면 옵션을 제시하고 재질문합니다.

### Q1. 범위 축 (Breadth)
> *"이 리팩터링은 어느 범위까지입니까?"*
- (a) **단일 파일** 내부만
- (b) **같은 디렉터리** 몇 개 파일
- (c) **하나의 모듈/기능** 단위
- (d) **여러 모듈에 걸친 cross-cutting**

→ (d)면 **사전 경고**: 반드시 여러 PR로 쪼갤 것.

### Q2. 깊이 축 (Depth)
> *"어느 계층까지 건드리나요?"*
- (a) **내부 구현만** (함수 분리, 변수명 등)
- (b) **타입/시그니처**까지 (내부 유지 + 타입 개선)
- (c) **Public API**(export, Props)까지 — **소비자 영향 있음**
- (d) **데이터 모델**(스토어 구조, API 응답)까지 — **광범위 영향**

→ (c), (d)면 **Blast Radius 분석 필수**. (d)는 별도 마이그레이션 계획 필요.

### Q3. 행동 변경 여부 (Behavior Change)
> *"동작이 바뀌어야 합니까?"*
- (a) **0% — 완전히 동일 동작**
- (b) **의도된 변경 포함** (버그 수정 등, **별도 커밋 분리 필요**)
- (c) **모호함** — 현재 버그가 있는지도 불명확

→ (a)면 TDD 스킬 불필요. Characterization Test로 고정.
→ (b)면 리팩터링 PR과 행동 변경 PR을 **반드시 분리**.
→ (c)면 먼저 동작 파악부터 (Phase 0).

### Q4. 시간 박스 (Time Box)
> *"이번 세션에서 어디까지 완료하길 원하시나요?"*
- (a) **한 PR 완료**까지 (Small 이하)
- (b) **계획 수립**까지 (여러 PR로 쪼개서 제안)
- (c) **첫 PR 실행**만 (나머지는 사용자가 이어감)
- (d) **탐색적 작업** — 범위 자체를 확정하는 것이 목표

→ (a)가 기본. 커져 보이면 (b)로 전환 제안.

### Q5. 리스크 허용 (Risk Tolerance)
> *"이 영역에 대한 안전 요구 수준은?"*
- (a) **프로덕션 크리티컬** — 테스트 100% 보장 필요
- (b) **일반** — 타입체크 + 기존 테스트 GREEN
- (c) **실험 영역** — 빠르게 반복, 롤백 용이

→ (a)면 Characterization Test + Smoke Test + Manual QA 필수.

### Q6. 테스트 포함 여부
> *"이 PR에 테스트를 포함합니까?"*
- (a) 기존 테스트 **GREEN 유지만**
- (b) **Characterization Test 추가** (현재 동작 고정)
- (c) **신규 테스트 추가** (개선된 설계 기반)
- (d) 테스트 없음 — **smoke test로만 검증**

### Q7. Scope 외 이슈 정책
> *"작업 중 Scope 밖 이슈를 발견하면?"*
- (a) **무조건 기록만** — 이번 PR에 미포함 (권장)
- (b) **한 줄짜리 명백한 수정**은 별도 커밋으로 포함
- (c) **관련 있으면** 같은 PR에 포함

→ (a)가 기본. (c)는 PR 비대화 위험 큼.

### Q8. 완료 기준 (Definition of Done)
> *"이 PR이 '완료'되었다는 기준은?"*
- 타입체크 통과
- 기존 테스트 GREEN
- 새 테스트 작성·통과
- Playwright smoke 통과
- 수동 QA 체크리스트
- 팀 리뷰 승인

여러 개 선택. 모호하면 `tsc + tests` 기본.

## Follow-up Questions (유형별)

### 레거시 정리성 리팩터링
- *"여러 저자의 스타일이 섞여 있나요? 어떤 스타일을 기준으로 통일합니까?"*
- *"라이브러리 공식 패턴을 따르길 원하시나요, 프로젝트 기존 패턴을 유지하시나요?"*

### 마이그레이션 (A → B)
- *"전환 중간 상태(A와 B 공존)를 허용하나요?"*
- *"Adapter 계층을 둘까요, 한 번에 교체할까요?"*
- *"A를 먼저 deprecate 표시할까요?"*

### 성능 리팩터링
- *"성능 목표 수치가 있나요?"* (예: LCP < 2.5s, 특정 렌더 < 16ms)
- *"측정 도구는 Chrome DevTools / React Profiler / Lighthouse 중 무엇?"*

### 접근성 리팩터링
- *"WCAG 2.1 AA 기준으로 진행하나요?"*
- *"키보드 네비게이션 테스트 포함합니까?"*

## 답변이 불충분할 때

### 패턴 1 — 옵션 제시
> *"Q3 (행동 변경)에 답변이 없습니다. 아래 중 선택해 주세요:*
> *(a) 완전히 동일 동작*
> *(b) 의도된 변경 포함*
> *(c) 모호 — 먼저 파악 필요"*

### 패턴 2 — 가정 공개
> *"명시 없으므로 다음을 가정합니다. 틀리면 알려주세요:*
> *- 범위: 단일 파일*
> *- 행동 변경: 0%*
> *- 완료 기준: tsc + 기존 테스트 GREEN"*

### 패턴 3 — 단계 분리 제안
> *"Scope가 너무 넓어 보입니다. 다음으로 쪼개는 건 어떨까요?*
> *PR 1: 타입 개선 (5 파일)*
> *PR 2: 함수 분리 (3 파일)*
> *PR 3: Public API 변경 (소비자 포함, 12 파일)"*

## 금지 사항

AI 에이전트가 Scope Narrowing 없이 하지 말아야 할 것:

- ❌ "일반적으로 이 경우엔..." 식의 가정
- ❌ 질문 없이 바로 코드 수정
- ❌ 8개 질문을 한 번에 모두 던져 사용자 부담 주기 (3~4개씩 묶어서)
- ❌ "전부 리팩터링하겠습니다" 식의 무한 Scope
- ❌ 답변 무시하고 다른 범위로 작업
