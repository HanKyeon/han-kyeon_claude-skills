# Unknown-Answer Playbook

모든 메타-스킬 인터뷰에서 **"모르겠음" (z) 옵션을 기본 탑재**하고, 선택 시 일관된 3단계 처리를 수행하도록 하는 공통 protocol입니다.

## 왜 필요한가

사용자가 질문에 답을 못 할 때 기존 스킬들은 서로 다르게 대응했습니다:
- `asset-factory` — (c) 모르겠음 + 힌트 제공 (잘 됨)
- `skill-author` — F-series follow-ups가 있지만 "그래서 답을 모르면?" 명확한 경로 없음
- `cfh-harness` — 옵션 제시 없이 자유 서술 기대 → 사용자 이탈
- `tdd-first`·`refactoring-strategy` — 일부 질문에만 fallback 존재

이 playbook은 **모든 메타-스킬 질문에 (z) 모르겠음을 붙이고 동일 처리 흐름**을 적용합니다.

## 공통 처리 3단계

### Step 1 — 구체 예시 2~3개 제시

```
(z) 모르겠음을 선택하셨습니다. 비슷한 상황의 예시를 드릴게요:

예시 A — <상황 묘사 1~2줄> → 이 경우라면 (a)
예시 B — <상황 묘사 1~2줄> → 이 경우라면 (b)
예시 C — <상황 묘사 1~2줄> → 이 경우라면 (c)

지금 하려는 작업이 A/B/C 중 어느 쪽에 더 가깝나요?
```

예시는 **질문의 맥락에 맞춰 구체적으로** 작성. 추상적 예시("소규모 프로젝트 vs 대규모")는 금지 — 사용자가 자기 상황을 대입하기 어려움.

### Step 2 — 그래도 모르면 "안전한 기본값 + 수정 가능 안내"

```
판단이 어려우시면 안전한 기본값 (<기본값>)으로 진행하겠습니다.
나중에 필요 시 <방법>으로 수정하실 수 있습니다.

계속 (<기본값>)으로 진행할까요? [y/n]
```

안전한 기본값 선택 원칙:
- **되돌리기 쉬운 쪽** (최소 권한·최소 에이전트·기본 패턴)
- **사용자가 이탈할 가능성이 낮은 쪽**
- **명시적 재검토가 쉬운 쪽** (프로젝트 로컬 `./.claude/`가 전역 `~/.claude`보다 수정 쉬움)

### Step 3 — y면 진행, n이면 질문 보류

- `y` → 기본값으로 해당 질문 확정, 다음 Phase 진행
- `n` → "잠시 중단하고 조사하신 뒤 다시 `/cfh-...`로 돌아오시면 됩니다" 안내

## 스킬별 기본값 레퍼런스

### skill-author

| 질문 | 기본값 | 근거 |
|---|---|---|
| Q1 한 문장 목적 | (z) 불가 — 목적이 없으면 스킬 작성 중단 | 목적 필수 |
| Q2 트리거 키워드 | 사용자가 던진 `$ARGUMENTS`에서 토큰 추출 후 제안 | 최소 트리거 유지 |
| Q3 반-트리거 | "유사 스킬 이름" 1개 자동 삽입 | 충돌 방지 최소 |
| Q4 핵심 원칙 | "프로젝트 컨벤션 준수"·"라이브러리 공식 문서 우선" 기본 2개 | 모든 스킬 공통 기초 |
| Q5 출력 형태 | "질문 리스트 형식" | 가장 일반적 |
| Q6 참조 자료 | "없음 (SKILL.md 단일 파일)" | 처음부터 references 분리는 over-engineering |

### cfh-harness

| 질문 | 기본값 | 근거 |
|---|---|---|
| Q1 태스크 성격 | (c) 다축 평가 | 가장 일반적 PR 리뷰 케이스 |
| Q2 입출력 | "입력: 파일 세트, 출력: 리포트" | Expert Pool 표준 |
| Q3 전문성 축 | "convention / logic / test" 3축 | 최소 구성 |
| Q4 실패 비용 | (b) 중간 | 보수적 기본 |
| Q5 규모 | 3 에이전트 | 작게 시작 |
| Q6 관측성 | (c) CI 연동 | 자동 감지 경로 확보 |

### asset-factory

| 질문 | 기본값 | 근거 |
|---|---|---|
| Q1 반복 가능? | (b) 한 번만 → 생성 중단 | over-build 방지 |
| Q2 협업 필요? | (b) 아니오 → Q3 | 팀 설계는 명시 의사 있을 때만 |
| Q3 트리거 방식 | (b) 명시 호출 → command | skill 생성보다 부담 낮음 |

### tdd-first

| 질문 | 기본값 | 근거 |
|---|---|---|
| Phase 0 작업 단위 | "단일 함수" | 가장 좁게 |
| Phase 0 커버리지 목표 | "Happy + 명시적 Edge 1~2개" | 중간 수준 |
| Phase 1 Intent Q5 Out of Scope | "없음" | 사용자 직접 추가 가능 |

### refactoring-strategy

| 질문 | 기본값 | 근거 |
|---|---|---|
| Scope Q1 범위 | "단일 파일" | 가장 좁게 |
| Scope Q3 행동 변경 | "행동 보존" | 리팩터 정의 |
| Scope Q7 범위 외 이슈 | "Issue 트래커 분리" | TODO 주석 지양 |

## 활성화 규칙 (각 스킬 SKILL.md에 추가)

각 메타-스킬의 `## 활성화 시 반드시` 섹션에 아래 한 줄 추가:

```markdown
N. **(z) 모르겠음 fallback.** 모든 인터뷰 질문에 `(z) 모르겠음` 옵션 기본 탑재. 선택 시 `references/unknown-answer-playbook.md` (또는 `~/.claude/skills/asset-factory/references/unknown-answer-playbook.md`)의 3단계 처리 — 예시 2~3개 → 안전한 기본값 제안 → y/n.
```

## 하지 말 것

- 예시 제시 없이 바로 기본값 안내 — 사용자가 자기 상황 판단 못 함
- 기본값을 "임의로 일단 진행" 하고 나중에 공개 — 투명성 원칙 위반
- "다시 시작하세요" 같은 무력 안내 — 이탈 유도. 반드시 "진행 or 중단" 2지선 제시
