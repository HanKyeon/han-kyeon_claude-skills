# Intent Interview Template

Phase 1에서 AI가 사용자에게 던지는 **구조화된 질문 시퀀스**입니다. 이 질문들이 제대로 답변되면 Phase 2~4가 거의 자동으로 결정됩니다.

## 기본 6 질문 (모든 작업에 공통)

### Q1. 목표 (한 문장)
> *"이 함수/컴포넌트/기능이 하는 일을 **한 문장**으로 설명해 주세요."*

- 애매한 답이 오면 되물을 것: *"'처리한다'는 구체적으로 어떤 변환인가요?"*
- 한 문장이 불가능하면 **책임이 둘 이상**이라는 신호 → 분리 제안

### Q2. Happy Path 예시
> *"가장 일반적인 입력 1개와 기대 결과를 알려주세요."*

- 실제 값으로 받음 (`"foo"`, `[1, 2, 3]`, `{id: 1, name: 'X'}`).
- "대충 데이터"가 아닌 **구체 값**으로.

### Q3. Edge Cases
> *"아래 중 이 함수가 마주칠 가능성이 있는 입력은 어떤 건가요?"*
> - null / undefined
> - 빈 배열 / 빈 문자열 / 빈 객체
> - 0 / 음수 / 매우 큰 수
> - 중복 요소
> - 정렬되지 않은 순서
> - 순환 참조
> - 매우 긴 입력

각 선택에 대해 **"어떻게 동작해야 하는가"** 도 물어봄.

### Q4. Error Cases
> *"어떤 입력은 거부되어야 하나요? 거부할 때 어떻게 실패하나요?"*
> - throw Error
> - return null / undefined
> - return default value
> - 사용자에게 에러 메시지 표시

### Q5. Out of Scope
> *"이번 작업에서 **하지 않을 것**은 뭐죠?"*

- 이 질문이 **가장 중요**. Scope 폭발을 막는 핵심.
- 예: "지금은 로깅 안 합니다", "캐싱은 다음 단계", "i18n은 별도 작업"

### Q6. 관찰 방법
> *"이 동작을 어떻게 확인하나요? 아래 중 어떤 것이 검증 대상인가요?"*
> - 반환값
> - 호출한 콜백
> - DOM의 변화 (렌더 결과)
> - 전역 상태 (store) 변경
> - 네트워크 호출
> - 외부 side effect (localStorage 등)

**복수 선택 가능**. 이게 정해져야 테스트 작성 방식이 결정된다.

## 유형별 추가 질문

### 순수 함수 (pure function)
- Q7. 입력 타입과 반환 타입을 명시해 주세요.
- Q8. 동일 입력에 항상 동일 출력인가요? (랜덤/Date 의존?)

### React 컴포넌트
- Q7. Props 인터페이스는 어떻게 되나요?
- Q8. 렌더링할 주요 요소는? (button/form/table 등)
- Q9. 사용자 인터랙션이 있나요? (click/type/submit)
- Q10. 외부 데이터 의존? (props / context / store / query)
- Q11. 접근성 요구사항? (ARIA role, 키보드 네비)

### 커스텀 훅
- Q7. 입력(인자)과 반환값은?
- Q8. 내부에서 호출하는 다른 훅은?
- Q9. Side effect 있나요? (useEffect, subscribe)
- Q10. cleanup이 필요한가요?

### API 함수
- Q7. Endpoint와 method는?
- Q8. 요청 스키마 / 응답 스키마?
- Q9. 에러 응답 형식은?
- Q10. 재시도·타임아웃 정책은?

### 리팩터링 (동작 보존)
- Q7. 현재 동작을 설명해 주세요 (기대 동작이 아니라).
- Q8. 알려진 버그가 있나요? 이번에 고칠 건가요 별도인가요?
- Q9. Characterization Test가 있나요? 없으면 어떻게 기준선을 잡을까요?

## 답변이 불충분할 때

AI는 "알아서 가정하고 진행" 금지. 대신:

### 패턴 1 — 옵션 제시
> *"Edge case에 대한 답을 주지 않으셨습니다. 아래 중 선택해 주세요:*
> *(a) null 입력은 빈 배열을 반환*
> *(b) null 입력은 에러를 throw*
> *(c) null 입력은 불가능한 상황이다 (`!`로 단정)"*

### 패턴 2 — 기본값 제안 + 확인
> *"명시되지 않아서 다음을 가정하고 진행하겠습니다. 틀리면 알려주세요:*
> *- 빈 배열은 0 반환*
> *- 음수는 throw*"

### 패턴 3 — 코드에서 추측한 내용 공유
> *"기존 코드를 보니 이 함수의 호출부에서 null을 전달하는 경우가 없어 보입니다. 그래도 null 방어를 넣을까요?"*

## 질문 시점

- **새 기능**: Q1~Q6 전부 + 유형별 추가.
- **버그 수정**: Q1 (버그 증상) + Q3 (어떤 입력에서?) + Q4 (기대 vs 실제) + Q6 (재현 방법).
- **리팩터링**: Q1 (대상 범위) + Q7 (현재 동작) + Q5 (Out of Scope, 강하게).

## 안티패턴

AI가 하지 말아야 할 것:

- ❌ 질문 없이 곧장 코드 작성
- ❌ "일반적으로 이런 경우엔..."으로 가정 대체
- ❌ 질문을 한꺼번에 10개 던져 사용자 부담 주기 (한 번에 3~5개씩)
- ❌ 답변 안 읽고 기존 가정 유지
- ❌ "알아서 잘 처리하겠다" 식의 에스케이프
