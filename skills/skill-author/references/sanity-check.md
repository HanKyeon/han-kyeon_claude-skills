# Phase 1 Sanity Check

Phase 1 답변이 모두 모인 뒤, **요약 카드를 제시하기 직전** 수행하는 모순 감지 단계입니다. 사용자가 빠른 승인을 하기 전에 Claude가 답변끼리의 상충을 짚어 재확인받는 것이 목적입니다.

## 원칙

1. **모순만 지적.** 선호의 차이는 지적하지 말 것. 객관적으로 "동시에 성립하기 어려운 답변"만.
2. **해석 2개 제시.** 모순이 발견되면 "A로 볼 수도 있고 B로 볼 수도 있습니다. 어느 쪽이 맞습니까?" 형식으로.
3. **재질문은 최소.** 한 번의 재확인으로 충분하도록 관련 답변을 한 번에 묶어서 제시.
4. **통과도 명시.** 모순이 없으면 "Sanity check 통과" 한 줄로 사용자에게 알림 — 건너뛰었다는 인상 방지.

## 모순 감지 룰 6개

### R1. 목적 vs 반-트리거 충돌
- **조건**: Q1(한 문장 목적)에 포함된 핵심 명사가 Q3(반-트리거)에도 등장.
- **예**: Q1="테스트 작성 자동화", Q3="테스트만 수정하는 경우는 반-트리거" → 스킬이 떠야 하는지 떠서는 안 되는지 불분명.
- **재질문**: "Q1에서 '테스트'를 핵심 대상으로 두셨는데 Q3에서도 '테스트만 수정'을 반-트리거로 두셨습니다. 어떤 경우는 발동하고 어떤 경우는 안 발동하는지 한 문장으로 구분해 주시겠습니까?"

### R2. 트리거 범위 vs 키워드 수
- **조건**: Q2 키워드 2개 이하 + 목적(Q1)이 넓음.
- **예**: Q1="프론트엔드 전반의 품질 향상", Q2 키워드="lint, style" → 2 키워드로는 광범위 목적을 커버 못 함. 오발동 혹은 미발동 양쪽 위험.
- **재질문**: "목적이 '프론트엔드 전반'인데 키워드가 2개뿐입니다. 키워드를 더 수집하거나 목적을 좁히는 것 중 어느 쪽이 좋을까요?"

### R3. 핵심 원칙 vs 출력 형태 불일치
- **조건**: Q4 원칙에 "코드 수정"·"파일 생성" 있는데 Q5 출력이 "리포트만"·"질문 리스트만".
- **예**: Q4="테스트 파일을 자동 생성", Q5="리포트 형식의 제안" → 생성인지 제안인지 모호.
- **재질문**: "Q4에서 '자동 생성'을 원칙으로 두셨지만 Q5에서는 '제안'이라고 하셨습니다. Claude가 파일을 직접 쓰길 원하십니까, 제안만 내길 원하십니까?"

### R4. 원칙 수 vs 추상도
- **조건**: Q4 원칙이 3개 미만이거나, 원칙 중 하나 이상이 "좋은 코드를 쓴다" 같은 측정 불가 문구.
- **예**: Q4=["좋은 테스트를 쓴다", "사용자 행동 기반"] → 첫 번째가 측정 불가. 두 번째는 구체적.
- **재질문**: "'좋은 테스트'는 측정 기준이 필요합니다. 구체적으로 어떤 기준인가요? (예: getByRole 우선, 내부 상태 접근 금지)"

### R5. 기존 스킬과 트리거 겹침 (Pre-scan 결과 활용)
- **조건**: Pre-scan에서 발견된 기존 스킬 중 description 키워드가 Q2와 30% 이상 겹침.
- **예**: 기존 `tdd-first`의 키워드 {TDD, 테스트, 의도, writer}, 신규 스킬 Q2={TDD, 테스트, 커버리지}.
- **재질문**: "`tdd-first` 스킬이 이미 '테스트' 키워드를 담당합니다. 새 스킬은 (a) 그것과 병합 (b) 'TDD 이후 커버리지 보강' 같은 **뒤 단계**를 담당 (c) 전혀 다른 영역(Q3 반-트리거 명시 필요) 중 어느 쪽인가요?"

### R6. 범용성 vs 구체 파일 패턴
- **조건**: Q1 목적이 범용(예: "모든 PR에 적용")인데 Q2 파일 패턴이 좁음(예: `src/payments/**`).
- **예**: Q1="코드 품질 검증", Q2 파일 패턴="src/payments/**" → 결제 모듈에만 적용되는데 목적은 범용.
- **재질문**: "목적이 범용인데 파일 패턴이 결제에 한정됩니다. 범용 스킬인가요, 결제 도메인 스킬인가요?"

## 실행 순서

```
1. Phase 1의 Q1~Q6 답변 확보 완료
2. R1~R6 각각 체크
3. 모순 발견 건수:
   - 0건 → "Sanity check 통과" 표시 + 요약 카드 제시
   - 1건 → 해당 지점만 재질문, 답변 갱신 후 요약 카드
   - 2건 이상 → 모순 전부 한 번에 나열, 사용자가 일괄 답변, 갱신 후 재검사 (1회만 반복)
4. 2회째 검사 후에도 모순이 남으면 사용자에게 의도적으로 남긴 것인지 확인 → 승인받으면 그대로 Phase 2 진행
```

## 출력 템플릿

### 통과 시
```
✅ Sanity check 통과 — Phase 1 답변에 상충 없음. 요약 카드로 진행합니다.
```

### 모순 발견 시
```
⚠️ Sanity check — 아래 지점 재확인 부탁드립니다.

1. [R1 충돌] Q1에서 '<A>'를 대상으로 두셨는데 Q3에서 '<A의 일부>'를 반-트리거로 두셨습니다.
   → 어떤 경우는 발동, 어떤 경우는 안 발동인지 한 문장으로 구분해 주세요.

2. [R5 겹침] 기존 스킬 `<name>`의 키워드와 30% 겹칩니다.
   → (a) 병합 (b) 뒤 단계 담당 (c) 다른 영역 — 어느 쪽인가요?
```

## 하지 말 것

- 모순이 아닌 **보완 제안**을 이 단계에서 내지 말 것 (그건 Phase 3 Outline에서).
- 사용자의 선호를 추측해서 자동 수정하지 말 것 — 반드시 질문으로.
- 6개 룰을 모두 사용자에게 설명하지 말 것 — 내부 체크리스트로만 사용하고, 걸린 항목만 노출.
