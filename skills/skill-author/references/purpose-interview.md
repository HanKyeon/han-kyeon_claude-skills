# Purpose Interview — Phase 1 상세

기본 6개 질문은 모든 스킬 작성에 공통. 각 질문 아래의 **조건부 follow-up**은 답변이 특정 조건에 걸릴 때만 추가로 던집니다. decision tree 형태로 깊이를 동적으로 조절하는 것이 목적입니다.

## 원칙

1. **기본 6개 먼저.** 조건부 follow-up은 사용자가 기본 답변을 마친 뒤에만.
2. **한 번에 3개까지.** follow-up이 4개 이상 걸리면 과다 — 상위 2~3개만 던지고 나머지는 Phase 3에서.
3. **옵션 제시형 재질문.** 답변이 불완전해도 가정·추정하지 말고 (a)/(b)/(c) 옵션으로 재질문.
4. **답변 승인 후 저장.** 각 follow-up의 답변도 Phase 1 답변 번들에 포함되어 요약 카드에 표시.

---

## Q1. 한 문장 목적

> *"이 스킬은 무엇을 해결합니까? 한 문장으로 답해 주세요."*

### 좋은 답변
"React 컴포넌트를 렌더링하는 테스트를 RTL 관례에 맞춰 작성하도록 Claude를 유도한다."

### 약한 답변
"테스트 도와줌" (너무 광범위 → SKILL이 무엇을 하는지 Claude가 모름)

### 답변 해석
- 한 문장에 **대상·동작·기준**이 모두 없으면 재질문.
- "모든 것을 다 한다"는 답변은 **스킬 2개 이상으로 분리** 제안.

### 조건부 follow-up

- **F1a** (목적에 "자동화"·"생성"·"수정" 단어 포함) → "Claude가 파일을 직접 쓰는 스킬입니까, 제안만 내는 스킬입니까? (쓰기: Write 도구 필요 / 제안: 질문·리포트만)"
- **F1b** (목적이 "~에 관한 규칙" 형식) → "규칙을 **적용**하는 스킬인가요, 규칙을 **검증**하는 스킬인가요? (적용: 수정 제안 / 검증: 위반 리포트)"
- **F1c** (목적이 기존 스킬과 유사) → Pre-scan에서 발견된 인접 스킬을 제시하고 "이 스킬과 무엇이 다릅니까? 한 문장으로."

## Q2. 트리거 상황

> *"사용자가 어떤 단어·의도·파일 패턴에서 이 스킬이 떠야 합니까?"*

아래 4축으로 수집:

1. **키워드** — 사용자가 쓸 법한 단어·약어 (예: "TDD", "테스트 먼저", "red-green")
2. **사용자 의도** — 한 줄 의도문 (예: "새 기능을 안전하게 추가하고 싶다")
3. **파일 패턴** — `*.test.ts`, `src/hooks/**/*.ts` 등
4. **명시 호출** — `/<command>` 슬래시 커맨드와 짝을 이룰지

### 답변 해석
- 키워드 3개 미만 → 더 수집.
- 트리거 10개 이상 → 오히려 오발동 위험. 반-트리거(Q3) 강화 예고.

### 조건부 follow-up

- **F2a** (키워드 2개 이하) → "키워드가 2개면 발동 범위가 매우 좁습니다. 다음 중 어느 시나리오에서도 떠야 합니까? (a) 사용자가 키워드 없이 의도만 말할 때 (b) 다른 언어로 말할 때 (c) 오타 있을 때"
- **F2b** (키워드에 일반 명사 포함: "code", "file", "test" 같은) → "'<일반명사>'는 너무 흔해서 다른 스킬과 겹칠 가능성이 높습니다. 이 키워드를 (a) 빼기 (b) 조합어로 구체화('auth test' 같은) (c) 반-트리거에 주의 명시 중 어느 쪽으로?"
- **F2c** (파일 패턴 미지정) → "파일 경로가 주어졌을 때 발동 여부를 description에 반영할까요? 예: `src/features/payment/**` 편집 시 트리거. (y/n)"
- **F2d** (슬래시 커맨드 연동 여부 불명) → "`/cfh-새이름` 커맨드와 짝지어 수동으로도 호출할 수 있게 만들까요? 자동 트리거만으로 충분한가요?"

## Q3. 반-트리거

**중요도 높음.** 많은 스킬이 이 질문을 건너뛰어서 오발동합니다.

예:
- `tdd-first`의 반-트리거: "기존 코드 그대로 두고 테스트만 보강" → 그 경우는 `/cfh-tc` 커맨드.
- `refactoring-strategy`의 반-트리거: "버그 수정" (동작 변경이 목적) → 일반 수정 워크플로.

### 답변 해석
- 반-트리거를 못 쓰는 경우 → 트리거가 너무 범용. 다시 Q1~Q2로.

### 조건부 follow-up

- **F3a** (반-트리거 "없음"/"모름" 답변) → "혹시 아래 중 반-트리거로 둘 만한 것이 있습니까? (a) 비슷한 단어지만 다른 도메인 (b) 이미 다른 스킬이 담당하는 영역 (c) 스킬 범위 밖인 일반 요청 중 어느 것이든."
- **F3b** (Pre-scan에서 인접 스킬 발견됨) → "기존 `<name>` 스킬이 '<키워드>' 영역을 담당합니다. 새 스킬에서 이 영역은 반-트리거로 두는 게 맞죠? (y/n)"
- **F3c** (Q2 키워드가 10개 이상) → "트리거 키워드가 많아 오발동 위험이 있습니다. 반-트리거를 2~3개 더 추가할 만한 상황이 떠오르시나요?"

## Q4. 핵심 원칙

사용자가 "이 스킬이 떴을 때 Claude가 반드시 지켜야 할 규칙" 3~5개를 나열.

### 좋은 예
1. 사용자에게 Scope를 먼저 질문한다
2. 공식 문서 링크 없는 라이브러리 조언은 거부한다
3. 테스트 없는 영역은 리팩터링 전에 테스트부터 제안한다

### 나쁜 예
1. 좋은 코드를 쓴다 (❌ 측정 불가)
2. TDD를 한다 (❌ 그 자체가 스킬이라 순환)

### 조건부 follow-up

- **F4a** (원칙에 "좋은"·"올바른"·"적절한" 같은 주관어) → "<원칙 N>을 측정 가능한 형태로 바꿀 수 있을까요? 예: '좋은 테스트' → 'getByRole 우선 사용, 내부 상태 접근 금지'"
- **F4b** (원칙 3개 미만) → "원칙이 적으면 Claude가 자율적으로 해석할 여지가 큽니다. (a) 이대로 진행 (b) 2개 더 추가 중 어느 쪽?"
- **F4c** (Pre-scan에서 `CLAUDE.md` 발견) → "`CLAUDE.md`에 관련 규칙이 있습니다. 그중 <N>개를 원칙으로 가져올까요? 사용자 승인 후 반영합니다."

---

## Q5. 출력 형태

가능한 카테고리:
- **질문 리스트** (예: Scope Narrowing 프로토콜)
- **워크플로 단계** (예: 5 Phase TDD)
- **코드 변경 제안** (diff 형식)
- **리포트** (예: 리뷰 결과 REVIEW.md)
- **의사결정 가이드** (예: "A vs B 중 어느 패턴")

### 답변 해석
- 카테고리 불명확 → SKILL.md 포맷 결정이 어려움. 구체 예시 요청.

### 조건부 follow-up

- **F5a** (출력이 "리포트" 형식) → "리포트 파일명이 고정입니까? (예: 항상 `REVIEW.md`) 아니면 Claude가 매번 제안? 고정이면 어떤 이름?"
- **F5b** (출력이 "코드 변경") → "Claude가 Edit/Write 도구로 직접 수정합니까, diff 블록으로 보여준 뒤 사용자 승인 시에만 적용합니까?"
- **F5c** (출력이 "워크플로") → "각 단계 사이에 사용자 승인 대기가 필요합니까, 연속 실행입니까?"

## Q6. 참조 자료

SKILL.md에 **항상 로드되면 비용**인 긴 자료:
- 라이브러리 공식 안티패턴 링크 모음 → `references/library-anti-patterns.md`
- 체크리스트 (20 항목 이상) → `references/checklist.md`
- 긴 예시 코드 → `references/examples/`

### 답변 해석
- "SKILL.md 본문에 다 쓸 것" → 150줄 넘으면 반드시 분리.
- "참조 자료 없음" → OK. 짧은 스킬은 SKILL.md 하나로 충분.

### 조건부 follow-up

- **F6a** (예상 SKILL.md 길이가 150줄+) → "본문이 길어질 것 같습니다. (a) 섹션별 references 분리 (b) SKILL.md에 요약만 + 상세는 references (c) 그대로 한 파일로 중 어느 쪽?"
- **F6b** (참조 자료가 10개+) → "참조 자료가 많습니다. references/ 하위를 더 분리할까요? (예: `references/patterns/`, `references/checklists/`)"

## Phase 1 종료 — Sanity Check 호출

Q1~Q6의 답변과 조건부 follow-up 답변이 모두 모이면 `sanity-check.md`의 R1~R6 룰로 자동 검사를 수행. 모순이 있으면 재확인 → 없으면 요약 카드.

```
=== 스킬 요약 (초안) ===
이름: <name>
목적: <Q1 + F1 답변 통합>
트리거: <Q2 키워드·패턴·슬래시 커맨드 통합>
반-트리거: <Q3 + F3 답변 통합>
핵심 원칙: <Q4 + F4 답변 통합>
출력: <Q5 + F5 답변 통합>
참조: <Q6 + F6 답변 통합>

Sanity check: ✅ 통과 (또는 ⚠️ N건 재확인 완료)

이대로 Phase 2(트리거 초안 작성)로 진행해도 될까요?
```
